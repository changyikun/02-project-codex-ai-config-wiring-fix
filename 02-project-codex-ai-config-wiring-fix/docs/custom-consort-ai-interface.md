# 自定义剧情妃与 AI 接口规则

## 一、定位

该系统用于让玩家在不改底层代码的前提下，向当前后宫局中加入新的“剧情妃”。

原则只有两条：

- 数值、入宫合法性、位分与后续硬结果全部由系统规则判定。
- AI 只负责把玩家的描述整理成人设草案，并生成入宫通报、初见剧情、后续互动对白。

自定义剧情妃和普通妃嫔共用同一套底层字段，只额外增加“自定义来源”和“人设卡”两类元数据。

## 二、玩家入口

默认入口为：玩家在寝殿召见宫女娇娇，选择 `引荐新妃 / 新增剧情妃`。

提交方式支持两种：

1. 结构化填写
2. 自然语言描述

自然语言描述示例：

```txt
我想加一个新妃子，名字叫谢知微，性格温柔但野心重，家世是四品文官庶女，容貌高，心计高，药理低，和玩家初始中立。
```

## 三、系统字段

### 3.1 必填字段

| 字段 | 说明 | 校验规则 |
| --- | --- | --- |
| `name` | 角色姓名 | 同局唯一，长度 `2~8` 字 |
| `personality` | 性格标签 | 允许单标签或复合描述 |
| `familyBackground` | 家世背景 | 映射到初始位分范围、家族档位、部分特殊限制 |
| `aptitude` | 资质配置 | 可由玩家直接填写或由 AI 草案生成 |

### 3.2 可选字段

| 字段 | 说明 |
| --- | --- |
| `entrySource` | 入宫来源，可手填或自动推断 |
| `age` | 年龄，默认 `15~23` |
| `initialRelationToPlayer` | 对玩家初始好感 / 态度 |
| `initialRelationToEmperor` | 对皇帝初始真心倾向 |
| `bio` | 角色简介 |
| `portraitTags` | 立绘 / 文案风格标签 |

### 3.3 数值字段

自定义剧情妃可直接写入以下字段，但系统只做“范围校验”，不强制套用普通开局点数平衡。

| 字段 | 范围 |
| --- | --- |
| `health` | `0~1000` |
| `fortune` | `-100~100` |
| `ambition` | `-100~100` |
| `intrigue` | `0~1000` |
| `appearance` | `0~1000` |
| `temperament` | `0~1000` |
| `music` | `0~100` |
| `painting` | `0~100` |
| `needlework` | `0~100` |
| `pharmacology` | `0~100` |
| `politics` | `0~100` |

## 四、入宫流程

### 4.1 提交阶段

玩家通过娇娇提交草案后，系统生成一条 `pendingCustomConsort` 记录。

同一旬默认只允许新增 `1` 名待入宫剧情妃。  
全局默认自定义剧情妃上限为 `6` 名，该值应保留为可配置项。

### 4.2 次旬落地

下一旬开始时，由娇娇统一通报该角色入宫。  
系统根据 `familyBackground` 和 `entrySource` 选择合逻辑的入场方式。

默认映射如下：

| 家世 / 来源 | 默认入宫说明 |
| --- | --- |
| 官家女 | 家族举荐入宫 |
| 和亲 / 外族 | 和亲或外邦进献 |
| 商贾 / 低门第 | 家族献女、重金走动入宫 |
| 宫女出身 | 宫女被抬位 |
| 未指定 | 系统按家世自动补全 |

### 4.3 落地后效果

角色正式写入后宫角色池，并立即获得：

- 基础面板
- 位分与宫殿位置
- 与玩家、皇帝的初始关系
- 首次入宫剧情
- 后续可调用的 AI 人设卡

## 五、系统校验

系统在写入前必须校验以下内容：

1. 名称不重复
2. 年龄合法
3. 数值不越界
4. 家世与入宫来源不冲突
5. 初始位分不超过家世允许范围
6. 外族 / 和亲血脉限制会继续影响后续立储规则

若玩家手填内容不完整，则系统按家世模板补齐缺失字段。  
若玩家文本描述与硬规则冲突，则以硬规则校验结果为准，并由 AI 重新润色为合理表述。

## 六、AI 接口边界

### 6.1 AI 负责

- 把玩家自然语言整理成结构化草案
- 生成娇娇的次旬通报文本
- 生成入宫首秀剧情
- 生成人物长期说话风格、性格摘要、关系倾向

### 6.2 系统负责

- 最终字段校验
- 位分、属性、家世映射
- 是否成功入宫
- 何时入宫
- 与主系统的数值联动

### 6.3 推荐接口

| 接口 | 作用 | 返回 |
| --- | --- | --- |
| `custom-consort-draft` | 把玩家自由描述转为结构化草案 | JSON 草案 |
| `custom-consort-intro` | 生成次旬通报与初见剧情 | 文本 / 选项 |
| `custom-consort-dialogue` | 后续持续生成该角色对白 | 文本 / 选项 |

AI 不得直接返回最终数值结果并覆盖系统状态。  
所有 AI 返回内容都只能作为候选草案，由系统二次确认后落地。

## 七、与主链联动

自定义剧情妃落地后，应自动接入以下系统：

- 夜晚侍寝 / 陪伴竞争
- 宫斗与案件
- 怀孕与生育
- 位分与冷宫
- 家族与朝堂
- 皇嗣与立储
- 玩家感情攻略线

因此在数据结构上，建议为其增加以下元字段：

- `isCustomConsort = true`
- `customSource = player`
- `entrySource`
- `personaCard`
- `introReportDelivered`
- `insertedAtXun`

## 八、落地建议

正式写代码时，建议把该模块拆成：

1. `custom-consort-form`
2. `custom-consort-validate`
3. `custom-consort-queue`
4. `custom-consort-ai-bridge`
5. `custom-consort-insert`

这样可以保证：

- 玩家自由度足够高
- 数值系统仍然是唯一真相
- AI 只做剧情增强，不做硬判定
